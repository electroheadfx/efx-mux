<objective>
Close **Gap G-01 (HIGH)** primary symptom: user reverts a just-created (untracked) file via the Git sidebar → file is correctly removed from disk AND from the Git sidebar's Changes list → BUT the file-tree still renders the row. Subsequent delete attempt on the stale row errors because the file no longer exists.

**Hypothesis from 18-HUMAN-UAT.md:**
> git-status-changed → refreshTreePreservingState() either (a) is not emitted by revert_file_impl for WT_NEW paths, or (b) re-reads a stale list_directory result, or (c) the snapshot/restore path re-adds the now-deleted entry from the pre-revert snapshot.

**Root cause confirmed via code inspection (2026-04-17):**

1. **`src-tauri/src/git_ops.rs` revert_file_impl (lines 475-534)** executes `fs::remove_file` for WT_NEW OR shells out to `git checkout --` for WT_MODIFIED/WT_DELETED/WT_TYPECHANGE/WT_RENAMED. After the mutation succeeds, the function returns `Ok(())` **without calling `app.emit("git-status-changed", ())`**. Compare to `src-tauri/src/file_ops.rs:443` (`app.emit("git-status-changed", ());`) and `src-tauri/src/file_ops.rs:498` where mutating file-ops commands DO emit. **This is hypothesis (a). It is the primary root cause.**

2. **`src/components/git-control-tab.tsx` handleRevertFile (lines 214-225) and handleRevertAll (lines 227-259)** do not call `emit('git-status-changed')` on success either. They do call `refreshGitFiles()` which refreshes the Git sidebar by re-invoking `get_git_files`, but that does not trigger any file-tree listener.

3. `src/services/file-service.ts` lines 43/56/70/83/110/124 emit git-status-changed from 6 mutation wrappers (write_file_content, delete_file, rename_file, create_file, create_folder, copy_path), but `revertFile` is exposed from `git-service.ts` (NOT file-service.ts) and does NOT emit — consistent with the pattern that git-service calls do not emit (since they don't change the working tree in the same way). But revert IS a working-tree mutation. That pattern gap is the bug.

**Hypotheses (b) and (c) are ruled out:** `refreshTreePreservingState` re-reads `list_directory` via `initTree()` (file-tree.tsx:253), which invokes the Rust `list_directory` command that reads the actual filesystem — so the re-read is not stale. The snapshot only captures expanded folder paths and the selected entry's path (NOT the full entry list), so it cannot re-add a deleted entry. The bug is purely the missing emit.

**Fix (minimal, two-layer):**

- **Rust primary (authoritative):** revert_file_impl takes an `&AppHandle` parameter and calls `app.emit("git-status-changed", ())` after successful WT_NEW delete OR successful checkout. CURRENT / pure-INDEX paths do NOT emit (nothing mutated).
- **Frontend secondary (defense-in-depth, matches file-service.ts pattern):** handleRevertFile emits `git-status-changed` after `revertFile` resolves successfully. handleRevertAll emits once after the loop completes (regardless of partial failures — if any file was reverted, the tree needs to know).

Purpose: restore tree/editor-tab consistency after revert. Without this fix the user sees phantom rows and subsequent delete clicks fail with OS errors.

Output: 3 edits across 2 files — 2 Rust (signature + emit call), 1 frontend (double-emit in handleRevertFile + handleRevertAll), 1 new Rust test validating emit on WT_NEW path.
</objective>

```rust
pub fn revert_file_impl(repo_path: &str, file_path: &str) -> Result<(), GitError> {
    let repo = Repository::open(repo_path).map_err(|_| GitError::NotARepo)?;
    let workdir = repo.workdir().ok_or(GitError::NotARepo)?.to_path_buf();

    let rel_path = if Path::new(file_path).is_absolute() {
        Path::new(file_path)
            .strip_prefix(&workdir)
            .map_err(|_| GitError::FileNotFound)?
            .to_path_buf()
    } else {
        Path::new(file_path).to_path_buf()
    };
    let rel_str = rel_path.to_string_lossy().to_string();

    let status = repo
        .status_file(&rel_path)
        .map_err(|e| GitError::RevertError(format!("status_file failed: {}", e)))?;

    drop(repo);

    if status.contains(git2::Status::WT_NEW) {
        let abs = workdir.join(&rel_path);
        std::fs::remove_file(&abs)
            .map_err(|e| GitError::RevertError(format!("Failed to delete untracked file: {}", e)))?;
        return Ok(());  // <-- BUG: no emit before return
    }

    let needs_checkout = status.contains(git2::Status::WT_MODIFIED)
        || status.contains(git2::Status::WT_DELETED)
        || status.contains(git2::Status::WT_TYPECHANGE)
        || status.contains(git2::Status::WT_RENAMED);

    if !needs_checkout {
        return Ok(());  // no-op — correctly does not emit
    }

    let output = Command::new("git")
        .args(["checkout", "--", &rel_str])
        .current_dir(repo_path)
        .output()
        .map_err(|e| GitError::RevertError(format!("Failed to run git: {}", e)))?;

    if !output.status.success() {
        let stderr = String::from_utf8_lossy(&output.stderr).to_string();
        return Err(GitError::RevertError(stderr));
    }

    Ok(())  // <-- BUG: no emit after successful checkout
}

    #[tauri::command]
```
    ```rust
    pub async fn revert_file<R: Runtime>(
        repo_path: String,
        file_path: String,
        app: AppHandle<R>,

        Ok(())
    }
    ```

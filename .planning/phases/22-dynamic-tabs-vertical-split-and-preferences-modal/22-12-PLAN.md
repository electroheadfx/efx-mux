  <decision>Should the per-project tab counter be monotonic (current behavior) or gap-fill?</decision>
  <context>
UAT test 15 sub-issue (b): "User deletes Terminal-2 and adds a new one — it's named Terminal-3 instead of reusing 2. Counter is monotonic but user expects gap-fill (design question)."

The user explicitly flagged this as a design question and asked to confirm intent.

Background:
- Plan 22-01 introduced `allocateNextSessionName(project)` with a monotonic per-project counter (`projectTabCounter` signal). PTY session names follow `<project>`, `<project>-2`, `<project>-3`, ... so Terminal-N corresponds to tmux session `<project>-N` (with `-1` skipped — the bare name is N=1).
- D-12 (CONTEXT.md): "PTY session name is stable on scope move. No tmux rename-session on drag." The counter is the load-bearing piece keeping names stable.
- Phase 22 gap-closure plan 22-08 wires the counter to persist on every allocation.

PLANNER'S DEFENDED DEFAULT: keep monotonic.

Rationale for keeping monotonic:
1. **PTY safety**: A deleted Terminal-2 may have left tmux session `<project>-2` alive (kill_legacy_right_sessions handles right-only; main may persist). Re-using N=2 collides with that orphan session and re-attaches to it, surfacing stale screen content.
2. **Clipboard / muscle memory**: User may have copied content from Terminal-2 before deleting; recreating "Terminal-2" implies it's the same tab — confusing.
3. **Tmux history**: tmux preserves history per session-name. Re-creating Terminal-2 either reattaches to old buffer (jarring) or requires explicit history wipe (extra Rust call).
4. **Implementation simplicity**: Monotonic counter is one integer. Gap-fill requires scanning all live tab session names per allocation.

Rationale for gap-fill:
1. **User mental model**: "I closed tab 2, the next slot should be 2."
2. **Lower max number**: After many add/close cycles, monotonic counter grows unbounded; gap-fill stays small.
3. **UI consistency**: Visible tab labels stay sequential.

Recommendation: keep monotonic. Add a code comment + a `CHANGELOG` note explaining rationale.
  </context>
  <options>
    <option id="monotonic">
      <name>Keep monotonic (planner's defended default)</name>
      <pros>PTY-safe; no tmux history collision; matches D-12 stable-name principle; simpler implementation</pros>
      <cons>Long-running session can grow Terminal-12, Terminal-13, ... after many add/close cycles</cons>

Replace the body of `allocateNextSessionName` with a slot-scan implementation:
```ts
function allocateNextSessionName(project: string | null): { name: string; n: number } {
  const key = project ?? '';
  // Gather all in-use numeric suffixes across all 6 scopes
  const inUse = new Set<number>();
  for (const scope of ['main-0','main-1','main-2','right-0','right-1','right-2'] as const) {
    for (const tab of getTerminalScope(scope).tabs.value) {
      const m = /-(\d+)$/.exec(tab.sessionName);
      if (m) inUse.add(parseInt(m[1], 10));
      else inUse.add(1); // bare project name === N=1
    }
  }
  let n = 1;
  while (inUse.has(n)) n += 1;
  // Persist the max-in-use as a high-water mark to keep counter signal in sync
  const max = Math.max(0, ...inUse);
  projectTabCounter.value = new Map(projectTabCounter.value).set(key, max);
  if (key) void updateSession({ [`tab-counter:${key}`]: String(max) });
  const suffix = n > 1 ? String(n) : undefined;
  return { name: projectSessionName(project, suffix), n };
}
```

# Phase 19 — UI Design Contract

> Visual and interaction contract for the GSD Sub-Tabs pane (5 nested tabs under the right-panel `GSD` top tab). Generated by gsd-ui-researcher from CONTEXT.md (19-CONTEXT.md) + existing project tokens (`src/tokens.ts`). Verified by gsd-ui-checker.

## Design System

| Property | Value |
|----------|-------|
| Tool | none (custom Tailwind 4 + inline-style tokens — shadcn not applicable to Preact stack) |
| Preset | not applicable |
| Component library | none (hand-rolled Preact components; reuse `TabBar` and accordion pattern from `git-changes-tab.tsx`) |
| Icon library | `lucide-preact` ^1.8.0 (already installed) |
| Font | Geist (sans) / GeistMono (mono) — from `src/tokens.ts` `fonts` |

| Markdown fallback | `marked` ^14.1.4 (raw-render path on parse failure — D-16) |

All components are already internal primitives with a stable token system from Phase 10 ("Pixel-Perfect UI Rewrite"). New sub-tabs compose existing primitives (`TabBar`, accordion rows, cards) — no registry needed.

## Spacing Scale

Phase 19 uses the project's **existing** `spacing` token set from `src/tokens.ts`. This is a dense, desktop-grade 2-pt-base scale (pixel values) — intentionally tighter than 8-pt-web-scale because the entire app is a terminal-first workspace. Declared values (all even-pixel multiples):

| Token (tokens.ts key) | Value | Usage in Phase 19 |
|-----------------------|-------|-------------------|
| `spacing.none` | 0px | Reset internal padding |
| `spacing.xs` | 1px | Hairline alignment |
| `spacing.sm` | 2px | Icon-to-label inline gap inside sub-tab pills |
| `spacing.md` | 4px | Badge inner padding, compact row gaps |
| `spacing.lg` | 6px | Accordion row inner padding (match `git-changes-tab.tsx`) |
| `spacing.xl` | 8px | Sub-tab bar vertical padding (`TabBar` already uses this) |
| `spacing['3xl']` | 12px | Sub-tab pill horizontal padding (`TabBar` already uses this) |
| `spacing['4xl']` | 16px | State-tab card padding, Milestones-tab card inner padding |
| `spacing['5xl']` | 20px | Gap between State-tab section cards |

**Exceptions:** None. Phase 19 adds no new spacing values; it reuses the established Phase 10 scale. The executor MUST import from `tokens.ts` and MUST NOT introduce bespoke pixel values (e.g., no `24px`, `32px` unless composed as multiples of existing tokens).

**Row min-height for interactive rows:** 32px (accordion phase rows, milestone card headers) to match the established density from `git-changes-tab.tsx`.

## Typography

Phase 19 uses the project's **existing** `fontSizes` token set from `src/tokens.ts`. All values in px. The scale is deliberately small — desktop density. Two weights only: 400 (regular) and 600 (semibold for active + heading emphasis), matching `tab-bar.tsx` conventions.

| Role | tokens.ts key | Size (px) | Weight | Line Height | Font Family | Usage |
|------|---------------|-----------|--------|-------------|-------------|-------|
| Body / description text | `fontSizes.lg` | 13 | 400 | 1.5 | Geist | Milestone card description, accordion body text, State-tab section body |
| Table / row label | `fontSizes.base` | 12 | 400 | 1.4 | Geist | Progress-tab table cells, phase-row labels, list items |
| Sub-tab pill label | 11 (literal — matches existing `tab-bar.tsx`) | 11 | 400 inactive / 600 active | 1 | Geist | Sub-tab pill label (inherits from `TabBar`) |
| Section heading | `fontSizes.xl` | 15 | 600 | 1.3 | Geist | Card titles (milestone name, phase name when expanded, State-tab section titles e.g. "Decisions") |
| Display (status header) | `fontSizes['2xl']` | 20 | 600 | 1.2 | Geist | State-tab status-header card milestone name |
| Mono / code | `fontSizes.base` | 12 | 400 | 1.5 | GeistMono | YAML front-matter values (State tab), file paths in empty states, commit hashes / progress bar glyphs |

**Constraints:**
- No more than 4 distinct font-size tokens per sub-tab view.
- Only weights 400 and 600 are permitted. No 500, no 700.
- Line-height 1.5 for multi-line prose (body/description/mono); 1.2–1.3 for single-line titles and displays.
- Inline markdown from parser output (`**bold**`, `` `code` ``) inside History-tab entries must use weight 600 for bold and `fonts.mono` at 12px for inline code.

**Accent (`#258AD1`) is reserved EXCLUSIVELY for:**
1. Active sub-tab underline (`border-bottom: 2px solid colors.accent`) — delegated to `TabBar` (D-04).
2. Active sub-tab label weight emphasis — already `colors.textPrimary` + weight 600, NOT accent color. Accent is the underline only.
3. Active milestone card accent-border on the 🚧 in-progress milestone (`border-left: 2px solid colors.accent`).
4. Current-phase row left-edge indicator in Phases tab (2px left border when the phase matches STATE.md `milestone + current phase`).
5. Chevron icon (`ChevronRight` / `ChevronDown` from lucide-preact) on an **expanded** accordion row.
6. "Resume file" link in State-tab Session Continuity section (underline on hover).
7. `@@ hunk` headers in any preserved diff rendering (not expected in Phase 19, but documented for consistency with existing accordion conventions).

**Accent is NOT used for:**
- Regular text
- Card backgrounds
- All hover states (hover uses `colors.accentMuted` at 20% alpha only)
- Status badges (those use `statusGreen` / `statusYellow` / `textMuted`)
- Primary buttons (no buttons in Phase 19 — all sub-tabs are read-only per D-18)

**Destructive copy / color:** Not applicable. Phase 19 has zero destructive actions per D-18 (sub-tabs are read-only). No `diffRed` / `statusYellow` warnings for "dangerous" interactions because there are no interactions beyond click-to-expand.

## Copywriting Contract

Phase 19 has **no primary CTA** (all sub-tabs are read-only per D-18 — no buttons, no forms, no destructive actions). The copywriting contract therefore focuses on sub-tab labels, empty states, and error/fallback messages.

### Sub-tab pill labels (D-01)

Five pills, in this order, with Lucide icon to the left of each label. Icons chosen to match meaning without ambiguity.

| Order | Label | Lucide icon | Meaning |
|-------|-------|-------------|---------|
| 1 | `Milestones` | `Flag` | Release milestones list |
| 2 | `Phases` | `ListChecks` | Phase-level breakdown |
| 3 | `Progress` | `BarChart3` | Table + completion % |
| 4 | `History` | `History` | Shipped-milestone timeline |
| 5 | `State` | `Compass` | "Where am I?" current position (default tab per D-02) |

Labels are **sentence case, single word**. No punctuation. Do not pluralize differently ("Phase" / "Progresses"). Do not localize.

### Empty states (per-tab, when source section absent — D-16, D-20)

Tone: **helpful, operational, no apologies**. All empty states follow this pattern:

```
Heading:  {Section} not found
Body:     {Copy below}
Action:   [View raw {filename}]   ← text link, accent underline-on-hover
```

| Tab | Heading | Body | Action link |
|-----|---------|------|-------------|
| Milestones | `Milestones section not found` | `Expected ## Milestones in .planning/ROADMAP.md. View the raw file or run /gsd-new-project in a terminal to initialize.` | `View raw ROADMAP.md` |
| Phases | `Phase details not found` | `Expected ## Phase Details in .planning/ROADMAP.md. View the raw file to inspect current shape.` | `View raw ROADMAP.md` |
| Progress | `Progress table not found` | `Expected ## Progress in .planning/ROADMAP.md.` | `View raw ROADMAP.md` |
| History | `No shipped milestones yet` | `.planning/MILESTONES.md is empty or missing. History populates when /gsd-complete-milestone runs.` | `View raw MILESTONES.md` |
| State | `State file not found` | `Expected .planning/STATE.md. Run /gsd-new-project in a terminal to initialize.` | `View raw STATE.md` |

### Missing-file empty states (D-20 — whole-file absent)

When the source `.md` file itself does not exist (not just a section):

| Tab | Heading | Body |
|-----|---------|------|
| Milestones / Phases / Progress | `No .planning/ROADMAP.md found` | `Run /gsd-new-project in a terminal to initialize the planning directory.` |
| History | `No .planning/MILESTONES.md found` | `History populates automatically when /gsd-complete-milestone ships a milestone.` |
| State | `No .planning/STATE.md found` | `Run /gsd-new-project in a terminal to initialize the planning directory.` |

No "View raw" link in missing-file states (nothing to view).

### Error states (parse failure that is NOT an absent section — D-16)

Parse errors other than missing sections fall back to raw rendering silently (no error UI). `marked.parse` renders the whole file. If `marked` itself throws (extremely rare), the pane shows:

| Heading | Body |
|---------|------|
| `Unable to render {filename}` | `Parse failed. Open the file in an editor tab to inspect.` |

No technical error detail is surfaced (per established project tone — errors log to console via existing `console.warn('[efxmux] …')` convention).

### Destructive confirmations

None. Phase 19 has zero destructive actions (D-18).

### Status-badge labels (for screen readers / tooltip on hover)

| Glyph | Color | Tooltip text |
|-------|-------|--------------|
| `✓` | `colors.statusGreen` | `Complete` |
| `◆` | `colors.statusYellow` | `In progress` |
| `○` | `colors.textMuted` | `Not started` |
| `✅` | `colors.statusGreen` | `Shipped` |
| `🚧` | `colors.statusYellow` | `In progress` |

### Loading states

First-parse loading per tab is expected to be instant (<100ms for files under 100KB). No spinner. If a read takes >500ms (unlikely), show the text `Loading…` at 13px `colors.textMuted`, centered. No animated skeleton — too heavy for a read-only pane.

### Layout spec — per sub-tab

**Milestones tab (card grid):**
- 1 card per milestone.
- Card: `bgElevated` background, `bgBorder` 1px border, 6px radius (`radii.lg`), 16px inner padding, 20px vertical gap between cards.
- Header row: status glyph (20px) + milestone name (15px/600) + phase range (12px/400, `textMuted`).
- Body: ship date + phase/plan counts (12px/400, `textMuted`).
- In-progress milestone: 2px `colors.accent` left border (the ONE accent card).

**Phases tab (accordion list):**
- 1 collapsible row per phase.
- Row (collapsed): min-height 32px, 6px vertical padding, 8px horizontal.
  - Left: `ChevronRight` (14px, `textMuted`).
  - Then: status badge (`✓ / ◆ / ○`) + phase name (12px/400) + plan count (12px/400, `textMuted`, right-aligned).
- Row (expanded): `ChevronDown` swap; reveal panel below with `bgElevated` background, 16px padding:
  - Goal (13px/400)
  - Depends-on (12px/400, `textMuted`)
  - Requirements list (12px/400, comma-separated or bullet)
  - Plans list (12px/400, one per line, status glyph prefix)
  - Success criteria (12px/400, numbered)
- Current phase (matches STATE.md `milestone_name` + `stopped_at`): 2px `colors.accent` left border on the row.
- Expand animation: 150ms ease-out max-height (match existing Phase 10 conventions). No bounce.

**Progress tab (table):**
- Top: summary header card — `bgElevated`, 16px padding, shows `v0.3.0 Workspace Evolution · {X}/{Y} phases · {percent}%` (15px/600 + 12px/400 secondary).
- Table: 5 columns (Phase, Milestone, Plans, Status, Completed).
- Header row: 12px/600, `textSecondary`, `bgElevated` background.
- Body rows: 12px/400, `textPrimary`; alternate rows `bgElevated`/`bgBase`; dividers use `bgBorder`.
- Status cell: badge glyph + label (`Complete` / `In progress` / `Not started`).

**History tab (timeline):**
- 1 block per shipped milestone (from `MILESTONES.md`).
- Block: 16px vertical gap.
- Block header: status glyph + milestone title (15px/600) + ship date (12px/400, `textMuted`).
- Body: key-accomplishments list (13px/400, bullet), plan/phase/task counts (12px/400, `textMuted`).
- Inline markdown (`**bold**`, inline code) rendered from parser output; use weight 600 for bold, `GeistMono` 12px for code.

**State tab (ordered sections):**
- Section 1 — Status header card: `bgElevated`, 16px padding, 20px vertical gap below:
  - Milestone name 20px/600.
  - Row: status label (12px/400) · progress % (12px/400, `statusGreen` if 100, `statusYellow` if <100) · last-activity string (12px/400, `textMuted`).
- Sections 2–6 rendered as stacked cards (20px gap, `bgElevated`, 16px padding, 15px/600 title + 13px/400 body):
  - `Current Position`
  - `Decisions`  (full list, bullet)
  - `Pending Todos`
  - `Blockers / Concerns`
  - `Session Continuity`  (timestamp 12px mono, stopped-at, resume file as accent-underlined link)
- `## Quick Tasks Completed` is **explicitly excluded** per D-10.

## Accessibility

Minimal but non-negotiable requirements for Phase 19:

- Sub-tab pills: reuse `TabBar`'s existing `<button>` semantics (already accessible). No custom `aria-*` additions needed.
- Accordion rows (Phases tab): use `<button>` with `aria-expanded={true|false}` and `aria-controls="phase-{slug}-panel"`. Expanded panel has `id="phase-{slug}-panel"` and `role="region"`.
- Status glyphs (`✓ / ◆ / ○ / ✅ / 🚧`): wrap in `<span aria-label="{tooltip text from copywriting contract}">` so screen readers announce `Complete` / `In progress` / `Not started` / `Shipped`.
- Keyboard navigation: D-05 defers sub-tab keyboard shortcuts. Accordion rows remain keyboard-reachable via default Tab (inherits from button element) and toggle via Enter/Space (default button behavior — no extra code).
- Link semantics: "View raw {file}" empty-state action is a `<button>` styled as a link (accent underline on hover) — NOT an `<a href>` (no real URL). Uses `role="link"` only if native button semantics cause any confusion; default button semantics preferred.
- Color contrast: all text/background combos above are inherited from Phase 10's verified palette (textPrimary on bgBase = 13.7:1, textMuted on bgBase = 5.2:1 — both pass WCAG AA).

## State Machine

Click-only, read-only UI. State transitions per sub-tab:

```
Sub-tab selection  (global, persisted)
  Milestones ⇄ Phases ⇄ Progress ⇄ History ⇄ State
  default: State (D-02)

Phase accordion  (per phase, ephemeral — not persisted)
  collapsed ⇄ expanded
  trigger: click on row button OR Enter/Space when focused
  default: all collapsed on tab mount

Data refresh  (event-driven — D-19)
  idle → parsing → rendered
  trigger: md-file-changed event for any of the 3 source files
  cache: per-file, keyed by absolute path (D-17)
  invalidation: on file-change event for matching path

Empty state   (derived, not stored)
  → missing file: friendly missing-file copy + no link
  → missing section: friendly section-missing copy + "View raw" link
  → parse error: fallback to marked.js raw render silently
```

No component-local state crosses sub-tab boundaries except the accordion expand-set, which is per-tab and resets when the user leaves and returns to the Phases tab (ephemeral, per D-22 scroll reset spirit). Planner may decide to preserve the expand-set within a session if desired — that choice is Claude's Discretion per CONTEXT.md.

## Interaction Contract

| Trigger | Target | Effect |
|---------|--------|--------|
| Click sub-tab pill | `TabBar` | Update `gsdSubTab` signal; content area rerenders; scroll resets to top (D-22); persists to `AppState` (D-03) |
| Click phase row (Phases tab) | Accordion button | Toggle `aria-expanded`; animate max-height 150ms ease-out |
| Click "View raw {file}" | Empty-state link | Switch the current sub-tab to a raw-render view using `marked.parse` on the whole file; a small "← Back" button (12px, `colors.textMuted`, `ArrowLeft` icon) returns to the structured view |
| Click "Resume file" path in State tab | Session Continuity link | Open the referenced file in a main-panel editor tab (if file-service supports it) — planner to confirm; if not supported, copy path to clipboard and show toast `Path copied` |
| File change in `.planning/` | Tauri `md-file-changed` (or new `planning-file-changed`) | Invalidate parse cache for that path (D-17); re-parse; rerender active sub-tab if its source matches |
| Missing project / missing `.planning/` | Pane render | Show per-tab missing-file empty state (see copywriting) |

No drag-and-drop, no context menus, no input fields, no keyboard shortcuts (D-05), no destructive actions (D-18). The pane is strictly a viewer.

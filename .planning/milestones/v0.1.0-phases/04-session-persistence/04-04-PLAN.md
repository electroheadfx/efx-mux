  <action>
Two targeted fixes in src/theme/theme-manager.js:

**Fix 1 — initTheme() must call setThemeMode(savedMode) after applyTheme(theme)**

In `initTheme()` (around line 164), after `applyTheme(theme)` is called, add `setThemeMode(savedMode)`. This ensures that when savedMode is 'light', the inline CSS vars set by applyTheme() are cleared so that :root[data-theme="light"] in theme.css takes effect. Currently applyTheme() unconditionally sets dark theme inline CSS vars, and setThemeMode() is never called on startup.

The data-theme attribute is already set earlier in initTheme() (line 161), but setThemeMode() also handles clearing inline CSS properties for light mode (lines 96-100) which is the critical missing step.

Note: setThemeMode() calls persistThemeMode() internally, which will re-save the same mode to state.json. This is harmless (idempotent write of the same value) and simpler than adding a flag to skip persistence.

**Fix 2 — Replace persistent localStorage theme-manual flag with session-scoped variable**

The `toggleThemeMode()` function (line 122) sets `localStorage.setItem('efxmux:theme-manual', 'true')` which is never cleared. This permanently blocks the OS prefers-color-scheme listener in `initOsThemeListener()` (line 141 checks this flag).

Replace this with a module-scoped boolean variable:
1. Add `let manualToggle = false;` at module scope (near line 14, after `currentTheme`).
2. In `toggleThemeMode()`: replace `localStorage.setItem(...)` with `manualToggle = true;`.
3. In `initOsThemeListener()`: replace `localStorage.getItem('efxmux:theme-manual') !== 'true'` with `!manualToggle`.
4. Remove any remaining references to `efxmux:theme-manual` in localStorage.

This way, the manual-override flag resets on every app launch, allowing OS theme changes to work on fresh starts while still preventing OS overrides within the current session after a manual toggle.

    function resolveHeaderCreateTarget(): string {
      const project = getActiveProject();
      const projectPath = project?.path || '';
      // If selectedIndex points at something, resolve by kind
      let selectedEntry: FileEntry | undefined;
      if (viewMode.value === 'flat') {
        selectedEntry = entries.value[selectedIndex.value];

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User right-click / header button → FileTree | User-initiated interactions only. |
| FileTree → launch_external_editor IPC | App name (fixed literal from code — "Zed", "Visual Studio Code", etc.) and path (user's own project path). Path validated by Rust is_safe_path. |
| FileTree → open_default / reveal_in_finder IPC | Path only, same guard. |
| FileTree → detect_editors IPC | No arguments. Returns a plain boolean struct. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-18-04-01 | Injection | launch_external_editor app parameter | mitigate | `app` is a FIXED string literal from code ("Zed", "Visual Studio Code", "Cursor", "Sublime Text", "IntelliJ IDEA") — never user-typed. Even so, the Rust command passes it as a separate argv element to `open -a`. Impossible to inject shell metacharacters because JS never concatenates `app` with anything. |
| T-18-04-02 | Elevation of Privilege | Launching user's editor apps | accept | Reuses the Plan 18-01 threat disposition (T-18-01-06): editor apps run with user's privileges via LaunchServices. Acceptable trust boundary. |
| T-18-04-03 | Spoofing | Header Open In button targeting wrong project | mitigate | `getActiveProject()` resolves the current active project synchronously from `activeProjectName` signal. If the user switches projects between opening and clicking the menu, the stale closure still references the path captured at menu-build time — but this is the path the user visually associated with the button. Acceptable. |
| T-18-04-04 | Denial of Service | detect_editors runs 5 `which` spawns on every app load | mitigate | Cached in `detectedEditors` signal after first resolution. `ensureEditorsDetected()` uses an inflight guard to prevent concurrent duplicate calls. One-time cost on FileTree mount. |
| T-18-04-05 | Information Disclosure | Detected editors leak installed software | accept | Detection result is local-only (never emitted externally). Telling the user which apps are installed IS the feature. |
| T-18-04-06 | Tampering | User overrides macOS app registration via custom installed app with same name | accept | If the user has a malicious "Zed.app" registered with LaunchServices, it will launch instead of the real Zed — but this is the user's own choice. Efxmux cannot defend against an attacker who already has write access to the user's `/Applications` folder. Consistent with macOS trust model. |

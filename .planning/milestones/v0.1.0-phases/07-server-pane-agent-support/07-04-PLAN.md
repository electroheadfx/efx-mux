**Gap 1 -- ANSI colors not rendering (UAT test 4):**

Root cause: `ansiToHtml` in `src/server/ansi-html.ts` only handles basic ANSI codes (30-37, 90-97 foreground, code 1 bold). Modern dev servers (Vite, Next.js, etc.) emit 256-color (`38;5;N`) and truecolor (`38;2;R;G;B`) sequences. These multi-part codes are parsed as individual numbers, so code 38 is silently ignored and the span gets no color style, falling back to the container's inherited color.

Fix in `src/server/ansi-html.ts`:

1. Add a `color256(n: number): string` helper that returns hex color strings for the xterm-256color palette:
   - Colors 0-7: map to existing ANSI_COLORS (30+n)
   - Colors 8-15: map to existing ANSI_COLORS (90+(n-8))
   - Colors 16-231: 6x6x6 RGB cube. For color N: `r = Math.floor((N-16)/36)`, `g = Math.floor(((N-16)%36)/6)`, `b = (N-16)%6`. Each component maps: `0->0, 1->95, 2->135, 3->175, 4->215, 5->255`.
   - Colors 232-255: grayscale ramp. Value = `8 + (N-232) * 10`. Hex = that value for R, G, and B.

2. Rewrite the ANSI code processing loop to consume multi-part sequences correctly. Instead of iterating `for (const code of parts)`, use an index-based loop `for (let i = 0; i < parts.length; i++)` so it can advance past multi-part codes:
   - When `parts[i] === 38 && parts[i+1] === 5`: consume `parts[i+2]` as 256-color index, call `color256(parts[i+2])`, push `color:${hex}`, advance `i += 2`
   - When `parts[i] === 38 && parts[i+1] === 2`: consume `parts[i+2..i+4]` as R,G,B, push `color:#${hex}`, advance `i += 4`
   - When `parts[i] === 48 && parts[i+1] === 5`: same as above but `background-color`
   - When `parts[i] === 48 && parts[i+1] === 2`: same as above but `background-color`
   - Otherwise: existing logic for codes 0, 1, and ANSI_COLORS map

3. CSS belt-and-suspenders: In `src/styles/app.css`, change the `.server-pane-logs` rule to use `color: inherit` instead of `color: var(--color-text)`. This avoids any potential Tailwind 4 `@layer` specificity issues while still inheriting the correct default text color from the parent `.server-pane` element.

**Gap 3 -- Toolbar shows Start after restart (UAT test 6, MAJOR):**

Root cause: When `restartServer` is called, the Rust side kills the old process then spawns a new one. The old process's waiter thread emits `server-stopped` with exit code 143 AFTER the frontend has already set `serverStatus = 'running'` for the new process. The exit-143-is-clean fix from Task 1 partially addresses this, but a belt-and-suspenders `isRestarting` guard is needed for non-143 exit codes during restart.

Fix in `src/components/server-pane.tsx`:

1. Add a module-level mutable flag (NOT a signal, no re-render needed):
```typescript
let isRestarting = false;
```

2. In `handleRestart`, set the flag BEFORE calling `restartServer`:
```typescript
const handleRestart = async () => {
  const proj = getActiveProjectEntry();
  if (!proj?.server_cmd) return;
  isRestarting = true;
  serverLogs.value = [...serverLogs.value, ansiToHtml('[server] --- Restarting ---\n')];
  serverStatus.value = 'running';
  try {
    await restartServer(proj.server_cmd, proj.path);
  } catch (err) {
    serverLogs.value = [...serverLogs.value, ansiToHtml(`[server] Restart failed: ${err}\n`)];
    serverStatus.value = 'crashed';
  } finally {
    setTimeout(() => { isRestarting = false; }, 2000);
  }
};
```

3. In the `listenServerStopped` callback, skip state changes while restarting:
```typescript
listenServerStopped((exitCode: number) => {
  if (isRestarting) return;
  
  if (exitCode === 143 || exitCode === 137) {
    if (serverStatus.value === 'running') {
      serverStatus.value = 'stopped';
    }
    return;
  }
  if (exitCode >= 0 && serverStatus.value === 'running') {
    serverStatus.value = 'crashed';
    serverLogs.value = [...serverLogs.value, ansiToHtml(`[server] Process exited (code ${exitCode})\n`)];
  }
});
```

4. In `handleStop`, clear `isRestarting = false` to cancel any active restart guard.

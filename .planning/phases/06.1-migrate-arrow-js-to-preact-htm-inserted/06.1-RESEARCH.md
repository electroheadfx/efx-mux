# Phase 6.1: Migrate Arrow.js to Preact + Vite + TypeScript + Tailwind 4 - Research

**Researched:** 2026-04-08
**Domain:** Frontend framework migration (Arrow.js -> Preact), build toolchain (no-bundler -> Vite), styling (CSS -> Tailwind 4), type system (JS -> TypeScript)
**Confidence:** HIGH

## Summary

This phase is a big-bang frontend modernization: replacing Arrow.js with Preact + @preact/signals, introducing Vite as build tool and dev server, converting all JavaScript to TypeScript with JSX, and migrating CSS to Tailwind 4. The Rust backend is untouched -- only the frontend calling conventions change (from `window.__TAURI__` globals to `@tauri-apps/api` npm imports).

The migration is conceptually straightforward because Arrow.js `reactive()` maps directly to `@preact/signals` `signal()`, and Arrow.js `html` tagged templates map to JSX function components. The hardest parts are: (1) converting all inline styles in layout.css to Tailwind utility classes while preserving exact layout behavior, (2) ensuring xterm.js and other DOM-imperative code (drag-manager, terminal-manager) integrates correctly with Preact's virtual DOM, and (3) converting the CustomEvent-based cross-component communication to a signal-based or event-based pattern that works with Preact's render cycle.

**Primary recommendation:** Execute as a 3-wave migration: Wave 1 sets up Vite + Preact + Tailwind scaffolding with a single "hello world" component rendering; Wave 2 converts all components to TSX with Tailwind classes; Wave 3 removes Arrow.js vendor files and validates all behavior matches.

### Pattern 1: Arrow.js reactive() -> @preact/signals

**What:** Replace Arrow.js module-scoped `reactive({})` with exported signals.
**When to use:** Every component and the state-manager module.

```typescript
// BEFORE (Arrow.js)
import { reactive } from '@arrow-js/core';
const state = reactive({ visible: false, query: '', selectedIndex: 0 });
state.visible = true; // direct mutation triggers reactivity

// AFTER (Preact signals)
import { signal } from '@preact/signals';
const visible = signal(false);
const query = signal('');
const selectedIndex = signal(0);
visible.value = true; // .value access triggers reactivity
```
[CITED: preactjs.com/guide/v10/signals/]

### Pattern 2: Arrow.js html tagged template -> JSX function component

**What:** Replace `html\`...\`` templates with JSX function components.
**When to use:** Every component.

```typescript
// BEFORE (Arrow.js)
import { html } from '@arrow-js/core';
export const TabBar = (tabs, activeTab, onSwitch) => html`
  <div class="tab-bar">
    ${tabs.map(tab => html`
      <button
        class="${() => activeTab() === tab ? 'tab-btn active' : 'tab-btn'}"
        @click="${() => onSwitch(tab)}"
      >${tab}</button>
    `)}
  </div>
`;

// AFTER (Preact TSX)
import { Signal } from '@preact/signals';

interface TabBarProps {
  tabs: string[];
  activeTab: Signal<string>;
  onSwitch: (tab: string) => void;
}

export function TabBar({ tabs, activeTab, onSwitch }: TabBarProps) {
  return (
    <div class="flex gap-0.5 px-2 pt-1 bg-bg border-b border-border shrink-0">
      {tabs.map(tab => (
        <button
          class={activeTab.value === tab
            ? 'px-4 py-1.5 bg-bg-raised border-b-2 border-accent text-text-bright text-xs -mb-px'
            : 'px-4 py-1.5 bg-transparent border-b-2 border-transparent text-text text-xs -mb-px hover:bg-bg-raised'}
          onClick={() => onSwitch(tab)}
        >{tab}</button>
      ))}
    </div>
  );
}
```

### Pattern 4: DOM-imperative code in Preact (xterm.js, drag-manager)

**What:** Use Preact `useRef` + `useEffect` for DOM-imperative operations.
**When to use:** Terminal containers, drag handles, any code that directly manipulates DOM.

```typescript
import { useRef, useEffect } from 'preact/hooks';

function TerminalArea() {
  const containerRef = useRef<HTMLDivElement>(null);

  useEffect(() => {
    if (!containerRef.current) return;
    const { terminal, fitAddon } = createTerminal(containerRef.current, {});
    // ... connect PTY, attach resize handler
    return () => terminal.dispose();
  }, []);

  return <div ref={containerRef} class="flex-1 bg-bg overflow-hidden relative min-h-[100px] p-1" />;
}
```

### Pattern 5: CustomEvent communication -> signals or Preact context

**What:** Replace `document.dispatchEvent(new CustomEvent(...))` with shared signals.
**When to use:** Cross-component communication (project-changed, open-diff, file-opened).

```typescript
// Shared signal store (state-manager.ts)
import { signal } from '@preact/signals';

export const activeProject = signal<Project | null>(null);
export const rightTopTab = signal('GSD');

// Component reads signal directly -- no event listener needed
function RightPanel() {
  // Auto-rerenders when rightTopTab changes
  return <div>{rightTopTab.value === 'GSD' && <GSDViewer />}</div>;
}

// For events that trigger side effects (not just state), keep CustomEvent
// or use effect() to react to signal changes
```

### Anti-Patterns to Avoid
- **Mixing window.__TAURI__ and npm imports:** Pick one. With Vite, always use npm imports. Disable `withGlobalTauri` in tauri.conf.json. [CITED: v2.tauri.app/develop/calling-rust/]
- **Using className instead of class:** Preact supports `class` directly (unlike React). Use `class` for consistency with Tailwind. [ASSUMED]
- **Accessing .value in JSX text nodes:** For text content, you CAN pass the signal directly (`{count}` not `{count.value}`) for optimal performance -- Preact patches the DOM directly without re-rendering the component. But for conditionals and comparisons, you MUST use `.value`. [CITED: preactjs.com/guide/v10/signals/]
- **Forgetting to dispose effects:** `effect()` returns a dispose function. In Preact components, call it in useEffect cleanup.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Reactive state | Custom pub-sub or event bus | @preact/signals | Fine-grained reactivity, auto-tracking, no boilerplate |
| JSX transforms | Manual createElement calls | @preact/preset-vite | Handles JSX factory, HMR, devtools automatically |
| CSS utility generation | Manual utility classes | Tailwind 4 + @tailwindcss/vite | Automatic class detection, tree-shaking, @theme tokens |
| TypeScript compilation | Manual tsc pipeline | Vite (esbuild) | Sub-millisecond TS compilation, no separate build step |
| Tauri dev server | Custom HTTP server | Vite dev server | HMR, module resolution, asset handling built-in |

## Common Pitfalls

### Pitfall 1: Tailwind purging removes dynamic classes
**What goes wrong:** Tailwind scans source files for class names. If you construct class names dynamically (e.g., `bg-${color}`), they get purged.
**Why it happens:** Tailwind's JIT scanner uses static analysis, not runtime analysis.
**How to avoid:** Always use complete class strings. Use conditional ternaries with full class names, not string concatenation.
**Warning signs:** Missing styles in production build but visible in dev.
[CITED: tailwindcss.com/docs]

### Pitfall 2: xterm.js CSS must be imported separately
**What goes wrong:** xterm.js requires its own CSS file (`@xterm/xterm/css/xterm.css`). Tailwind won't generate these styles.
**Why it happens:** xterm.js uses its own CSS for terminal rendering, not Tailwind classes.
**How to avoid:** Import xterm.css in app.css: `@import "@xterm/xterm/css/xterm.css";` -- Vite will resolve the npm path.
**Warning signs:** Terminal renders but with broken layout/scrollbar.

### Pitfall 3: Preact signals .value in event handlers
**What goes wrong:** Forgetting `.value` when reading signals in event handlers or async code.
**Why it happens:** In JSX template expressions, Preact auto-unwraps signals. But in callbacks, closures, and async functions, you must use `.value` explicitly.
**How to avoid:** TypeScript will catch this -- signal type is `Signal<T>` not `T`.
**Warning signs:** TypeScript errors about Signal<T> not being assignable to T.

### Pitfall 4: Tauri CSP blocks Vite dev server
**What goes wrong:** Content Security Policy in tauri.conf.json blocks connections to localhost:5173 (Vite dev server) or inline styles.
**Why it happens:** Current CSP is `default-src 'self'; script-src 'self' 'unsafe-inline'; style-src 'self' 'unsafe-inline'`. Vite dev mode needs to connect to its own server.
**How to avoid:** In dev mode, Tauri uses `devUrl` which bypasses CSP for the main document. But Tailwind's runtime styles and Vite's HMR WebSocket need `'unsafe-inline'` for styles and `connect-src ws://localhost:*` for HMR. The existing CSP may need updating.
**Warning signs:** White screen in dev mode, console errors about CSP violations.

### Pitfall 5: Preact class vs className
**What goes wrong:** Using `className` out of React habit when Preact supports both but `class` is idiomatic.
**Why it happens:** React requires `className`; Preact accepts both `class` and `className`.
**How to avoid:** Use `class` consistently throughout to match Tailwind documentation patterns.
**Warning signs:** No functional issue, but inconsistency in codebase.

### Pitfall 6: Layout CSS with arbitrary values
**What goes wrong:** Some layout properties use CSS custom properties (--sidebar-w, --right-w) that are set dynamically by JavaScript. These don't map to static Tailwind classes.
**Why it happens:** Drag handles resize panels by setting CSS custom properties at runtime.
**How to avoid:** Keep CSS custom properties for dynamic layout dimensions. Use Tailwind for static styles. The `style` prop handles dynamic values: `style={{ width: 'var(--sidebar-w)' }}`.
**Warning signs:** Panel resizing breaks after Tailwind conversion.

## Open Questions

1. **Light mode implementation with Tailwind 4**
   - What we know: Current approach uses `[data-theme="light"]` CSS attribute selector to override CSS custom properties. Tailwind 4 has built-in dark mode support.
   - What's unclear: Whether to use Tailwind's `dark:` variant (which uses `prefers-color-scheme`) or keep the `data-theme` attribute approach for manual toggle.
   - Recommendation: Keep `data-theme` attribute approach with custom CSS overrides since the app supports manual dark/light toggle independent of OS preference (per THEME-04). Define both color sets in @theme and use `[data-theme="light"]` CSS rule to swap variables.

2. **CustomEvent migration completeness**
   - What we know: Current code uses 5+ CustomEvent types (project-changed, open-diff, file-opened, show-file-viewer, open-fuzzy-search, project-added).
   - What's unclear: Whether all of these should become signals or some should remain as events.
   - Recommendation: Convert state-representing events to signals (project-changed -> activeProject signal, open-diff -> rightTopTab signal). Keep action-triggering events as CustomEvent or simple callback props where the "event" triggers async work, not just state changes.

3. **drag-manager.js integration with Preact**
   - What we know: drag-manager.js is pure DOM manipulation -- it queries `[data-handle]` elements and attaches mouse event listeners. It sets CSS custom properties directly.
   - What's unclear: Whether Preact's virtual DOM will interfere with drag-manager's direct DOM queries.
   - Recommendation: Convert to .ts (TypeScript) but keep as framework-agnostic DOM code. Call initDragManager() after Preact's first render via requestAnimationFrame. The data-handle attributes are static, so Preact won't interfere.

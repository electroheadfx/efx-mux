<objective>
Close **Gap G-01 (HIGH)** secondary symptom: when a file is deleted (via revert-as-delete, the Delete key, or the context menu), if the file was open in an editor tab, that tab remains open pointing at a nonexistent path. The editor view continues to display stale content, and attempting to save produces a confusing error.

**Root cause from code inspection (src/components/editor-tab.tsx:89-113):**

```ts
// Re-read file content when git status changes (handles git checkout/revert)
useEffect(() => {
  let cancelled = false;
  const unlistenPromise = listen('git-status-changed', async () => {
    if (cancelled || !viewRef.current || !setupRef.current) return;
    try {
      const diskContent = await readFile(filePath);
      const savedContent = setupRef.current.getSavedContent();
      if (diskContent !== savedContent) {
        // File changed on disk -- update editor
        // ... update editor view ...
      }
    } catch {
      // File may have been deleted -- ignore    ← BUG: silently leaves tab open
    }
  });
  // ...
}, [filePath, tabId]);
```

The `catch {}` is the bug. It acknowledges the file may have been deleted and then DOES NOTHING. The user is left with a tab pointing at a vanished file. This compounds G-01: the user can't even click the stale tree row (G-01 primary), and if they had the file open, the tab stays too.

**Fix (minimal, reuses existing primitives):**

1. **New export in unified-tab-bar.tsx:** `closeEditorTabForDeletedFile(filePath: string): void` — finds the tab whose `filePath` matches, switches active tab to adjacent if needed, and REMOVES the tab from `editorTabs.value` WITHOUT going through `closeUnifiedTab` (which would surface the unsaved-changes confirm modal). The deleted-file case has no save path, so the modal is meaningless.

2. **Extend editor-tab.tsx catch block:** detect file-not-found from `readFile` errors and call `closeEditorTabForDeletedFile(filePath)`. File-service.ts readFile throws a `FileError` with a `code` field (pattern from other wrappers); the FS error for a missing file contains "No such file" on macOS. Match either pattern.

3. **file-service.ts readFile:** verify error format. If the current implementation does not wrap FS errors in a `FileError` with a stable code, add a `NotFound` / `ReadError` discriminant so the frontend can match reliably.

**Dependency on Plan 18-10:** 18-10 makes `revert_file` emit `git-status-changed`. Without that, the editor-tab listener never runs after a revert. Running 18-11 alone would cover only delete-via-Delete-key (which already emits via file-service.ts deleteFile). Running 18-10 then 18-11 covers the full Gap G-01 end-to-end.

Purpose: restore tab-to-disk consistency. No stale editor views. No confusing save errors.
Output: one new exported helper in unified-tab-bar.tsx; one catch-block extension in editor-tab.tsx; one error-shape refinement in file-service.ts; two new Vitest tests.

    ```ts
    /**
     * Plan 18-11 (Gap G-01 secondary fix): auto-close an editor tab when its
     * backing file is deleted from disk. Bypasses the unsaved-changes confirm
     * modal because the file is gone and saving is impossible. Called by
     * editor-tab.tsx when its git-status-changed listener detects that
     * readFile(filePath) now fails with a file-not-found error.
     *
     * If multiple tabs point at the same filePath (e.g. one pinned + one
     * unpinned), all matching tabs are removed. If filePath does not match
     * any open tab, this is a silent no-op.
     */
    export function closeEditorTabForDeletedFile(filePath: string): void {
      const matching = editorTabs.value.filter(t => t.filePath === filePath);
      if (matching.length === 0) return;

      // If any of the matching tabs is active, switch away first.
      const activeId = activeUnifiedTabId.value;
      if (matching.some(t => t.id === activeId)) {
        // Use the same helper closeUnifiedTab uses — switches to the adjacent
        // tab based on tabOrder. switchToAdjacentTab works only while the tab
        // is still present in allTabs, so call it BEFORE removing.
        switchToAdjacentTab(activeId);
      }

      // Clear dirty state for each matching tab so persistence doesn't keep a
      // dirty=true entry pointing at a deleted file.
      for (const tab of matching) {
        setEditorDirty(tab.id, false);
      }

      // Remove from editorTabs AND from tabOrder.
      const matchingIds = new Set(matching.map(t => t.id));
      setProjectEditorTabs(editorTabs.value.filter(t => !matchingIds.has(t.id)));
      setProjectTabOrder(tabOrder.value.filter(id => !matchingIds.has(id)));
    }
    ```

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-18-11-01 | Tampering | A malicious emit payload closes a tab the user wanted to keep | mitigate | The listener calls readFile(filePath) and checks the REAL error from the filesystem. An attacker cannot forge a "No such file" error unless they first delete the file, in which case the close is correct behavior. |
| T-18-11-02 | Denial of Service | Rapid consecutive git-status-changed events fire readFile N times per tab | accept | Each readFile call is non-blocking; Tokio spawn_blocking handles throughput. If readFile itself becomes a bottleneck, a debounce can be added later. No data loss risk. |
| T-18-11-03 | Information Disclosure | Error message substring match reveals file path in logs | n/a | filePath is already in the tab metadata and visible in the UI; logging the path is not a new disclosure. |
| T-18-11-04 | Spoofing | A sibling tab with a similar path is accidentally closed | mitigate | Match is exact-string equality on filePath (.filter(t => t.filePath === filePath)). No substring match, no glob. |
| T-18-11-05 | Elevation of Privilege | Bypass of unsaved-changes modal results in data loss | accept | The file is gone — saving is impossible. The user cannot recover the tab content by clicking "Save" because the target does not exist. Silently closing the tab is strictly better than leaving a ghost tab that errors on every save attempt. This is the intended behavior per G-01 "must-have truths". If a user wants content recovery, they must implement a separate "unsaved content vault" feature — out of scope. |

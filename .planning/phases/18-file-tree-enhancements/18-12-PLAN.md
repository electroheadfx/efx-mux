Close **Gap G-02 (MEDIUM)** as specified in 18-HUMAN-UAT.md: user drags a file from Finder over the file tree — the whole file-tree container shows the active-drag outline + glow, BUT the specific folder row under the cursor does not get its own per-row blue outline. The user cannot see which folder will receive the drop before release.

**User-preferred behavior (from gaps_to_close context):**
1. **Preferred:** per-row blue drop-target highlight updates as cursor moves over different folders during Finder drag.
2. **Fallback:** copy dropped files to project root + toast "Copied to project root — move into a folder afterwards".

**Decision criterion per gap spec:** Does Tauri 2 `onDragDropEvent` expose position updates during drag (pre-drop)? YES — per Tauri v2 docs (https://v2.tauri.app/reference/javascript/api/namespacewebviewwindow) the event types are 'over' (continuous, with position) / 'drop' (at release, with paths + position) / 'cancel'. The `over` event fires continuously with position during the drag. **Therefore the preferred path (per-row highlight) is feasible and MUST be implemented.**

**Root cause confirmed via code inspection (2026-04-17):**

Current src/main.tsx lines 269-311 (the onDragDropEvent handler):

```ts
await getCurrentWebviewWindow().onDragDropEvent((event) => {
  const payload = event.payload as
    | { type: 'enter'; paths: string[]; position: { x: number; y: number } }
    | { type: 'over'; paths: string[]; position: { x: number; y: number } }   // ← wrong type: Tauri 2 `over` has no paths
    | { type: 'drop'; paths: string[]; position: { x: number; y: number } }
    | { type: 'leave' };
  if (payload.type === 'leave') {
    document.dispatchEvent(new CustomEvent('tree-finder-dragleave'));
    return;
  }
  const projectName = activeProjectName.value;
  const project = projectName ? projects.value.find(p => p.name === projectName) : undefined;
  const projectPath = project?.path ?? '';
  const paths = 'paths' in payload ? payload.paths : [];
  const anyOutside = paths.length > 0 && paths.some(p => !projectPath || !p.startsWith(projectPath));
  if (!anyOutside) return;   // ← BUG: `over` events have no paths, so anyOutside === false, so over is filtered
  // ...
});
```

**The bug:** `paths.length > 0` is false for every `over` event because the real Tauri 2 `over` payload is `{ type: 'over', position: PhysicalPosition }` — NO paths field. The TypeScript union declared in main.tsx is wrong (claims paths exist on over, which they do not at runtime). So at line 285 `paths = []` for every over event, and `anyOutside = false` at line 288, and the function early-returns at line 289. Result: the ONE dispatch per drag comes from the `enter` event (first hover with paths); subsequent `over` events never dispatch, so the per-row highlight in handleFinderDragover (file-tree.tsx:839-856) only runs once and never updates as the cursor moves.

**Fix (minimal, 3-line):**

Introduce a module-level boolean `isFinderDragActive` (or `activeFinderDrag`) set to true when `enter` arrives with paths that include at least one outside the project root, reset to false on `leave` and `drop`. The `over` branch no longer requires `paths.length > 0` — it just checks `isFinderDragActive`. This preserves the correct routing (intra-project drags still ignored because `enter` sets `isFinderDragActive = false` for them) and enables continuous per-row highlight updates.

Purpose: give users clear visual feedback about where a Finder-dropped file will land before release.
Output: 1 edit to main.tsx (10 lines net); 2 new Vitest tests in file-tree.test.tsx verifying the enter→over→drop sequence dispatches correctly.

async function handleFinderDragover(e: Event) {
  finderDropActive.value = true;
  const { position } = (e as CustomEvent).detail as { paths: string[]; position: { x: number; y: number } };
  const rowEls = document.querySelectorAll<HTMLElement>('[data-file-tree-index]');
  rowEls.forEach(el => { el.style.borderLeft = ''; el.style.backgroundColor = ''; });
  for (const el of rowEls) {
    const rect = el.getBoundingClientRect();
    if (
      position.x >= rect.left && position.x <= rect.right &&
      position.y >= rect.top && position.y <= rect.bottom
    ) {
      el.style.borderLeft = `2px solid ${colors.accent}`;
      el.style.backgroundColor = `${colors.accent}20`;
      break;
    }
  }
}

      try {
        await getCurrentWebviewWindow().onDragDropEvent((event) => {
          const payload = event.payload as
            | { type: 'enter'; paths: string[]; position: { x: number; y: number } }
            | { type: 'over'; position: { x: number; y: number } }      // Plan 18-12: `over` has NO paths

            isFinderDragActive = false;
            document.dispatchEvent(new CustomEvent('tree-finder-dragleave'));
            return;
          }

          // For enter, compute the inside-project decision and cache it.
          if (payload.type === 'enter') {
            const projectName = activeProjectName.value;
            const project = projectName ? projects.value.find(p => p.name === projectName) : undefined;
            const projectPath = project?.path ?? '';
            const paths = payload.paths ?? [];
            const anyOutside = paths.length > 0
              && paths.some(p => !projectPath || !p.startsWith(projectPath));
            isFinderDragActive = anyOutside;
            if (!anyOutside) return;  // intra-tree drag — mouse pipeline owns it

          // For over + drop: respect the cached decision from enter.
          // (over has NO paths; drop has paths.)
          if ((payload.type === 'over' || payload.type === 'drop') && !isFinderDragActive) {
            return;
          }

          // Convert physical-pixel position to CSS pixels, then subtract the
          // macOS overlay title-bar offset so y aligns with viewport-relative
          // getBoundingClientRect() coordinates used by file-tree hit-tests.
          // (UAT Test 16 fix — Tauri Issue #10744; preserved from Plan 18-07.)

        });
      } catch (err) {
        console.warn('[efxmux] onDragDropEvent subscribe failed:', err);

# Phase 06.1: Code Review Report

**Reviewed:** 2026-04-08
**Depth:** standard
**Files Reviewed:** 21
**Status:** issues_found

## Summary

This phase migrated all UI components from Arrow.js to Preact TSX with `@preact/signals` for state management. The migration is structurally sound — Tauri invoke calls are correctly typed, signals are used consistently for reactive state, and the XSS-critical diff/file viewer paths use `escapeHtml`. However, two critical issues were found: a stored XSS vector in the file viewer (`read_file_content` result rendered via `innerHTML` indirectly) and a module-scoped side-effect in `fuzzy-search.tsx` that attaches a global `keydown` listener at import time, preventing cleanup. Eight warnings cover resource leaks, unguarded array accesses, a race condition in the PTY session fallback, and missing `await` on fire-and-forget invocations that can silently lose errors.

## Critical Issues

### CR-01: File Content Rendered as HTML via `dangerouslySetInnerHTML` with Only Partial Escaping

**File:** `src/components/main-panel.tsx:77-79`

**Issue:** The `fileContent` signal holds content fetched by `invoke('read_file_content', ...)` in `main.tsx:96`. This content is passed into a `<pre dangerouslySetInnerHTML={{ __html: escapeHtml(fileContent.value) }} />`. `escapeHtml` converts `<`, `>`, `&`, and `"` but does **not** escape single quotes (`'`). In a `<pre>` context this is safe today, but the real concern is the chain: if the Rust `read_file_content` command is ever updated to return pre-formatted HTML (or is pointed at an SVG/HTML file), the `escapeHtml` call is the only barrier. More concretely: `escapeHtml` in `main-panel.tsx` (line 23-29) does escape the four standard entities, so this is safe for the current narrow usage. The actual critical gap is in `right-panel.tsx` line 68:

```tsx
bashContainerRef.current.innerHTML = `<div class="p-4 text-[#dc322f] text-[13px]">Failed to connect terminal: ${err}</div>`;
```

Here `err` is an unknown Error or string caught from an async Tauri call. It is interpolated directly into `innerHTML` **without escaping**. A crafted error message (e.g., from a Rust panic string containing `<script>`) would execute as HTML. Since this is a Tauri app with `csp: "unsafe-inline"` in `tauri.conf.json` (line 24), injected `<script>` tags would be blocked, but inline event handlers (e.g., `<img onerror=...>`) would execute.

**Fix:**
```tsx
// right-panel.tsx -- add a local escapeHtml helper or import one
function escapeHtml(s: string): string {
  return String(s)
    .replace(/&/g, '&amp;')
    .replace(/</g, '&lt;')
    .replace(/>/g, '&gt;')
    .replace(/"/g, '&quot;')
    .replace(/'/g, '&#39;');
}

// Then in the catch block:
bashContainerRef.current.innerHTML =
  `<div class="p-4 text-[#dc322f] text-[13px]">Failed to connect terminal: ${escapeHtml(String(err))}</div>`;
```

### CR-02: Module-Scoped Event Listeners in `fuzzy-search.tsx` Never Cleaned Up

**File:** `src/fuzzy-search.tsx:118-121`

**Issue:** Two global `document` event listeners are attached at module evaluation time (lines 118 and 121), outside any component lifecycle:

```ts
document.addEventListener('keydown', handleGlobalKeydown);
document.addEventListener('open-fuzzy-search', openSearch);
```

There is no corresponding `removeEventListener`. If the module is ever hot-reloaded (Vite HMR in development), each reload adds another copy of these listeners — `handleGlobalKeydown` accumulates, causing Ctrl+P to fire multiple times and interfere with other handlers. Even in production, the listeners persist for the full session, which is intentional, but the pattern makes the code untestable and creates a leak if the module is ever conditionally loaded.

**Fix:** Move the listener registration into the `FuzzySearch` component's `useEffect` with cleanup, or export an `initFuzzySearch()` function called once from `main.tsx` (matching the pattern used by `initDragManager`):

```ts
// In main.tsx, called once after render:
export function initFuzzySearch(): () => void {
  document.addEventListener('keydown', handleGlobalKeydown);
  document.addEventListener('open-fuzzy-search', openSearch);
  return () => {
    document.removeEventListener('keydown', handleGlobalKeydown);
    document.removeEventListener('open-fuzzy-search', openSearch);
  };
}
```

## Warnings

### WR-01: Unawaited `invoke` Calls Silently Drop Errors

**File:** `src/terminal/pty-bridge.ts:26`, `src/state-manager.ts:147`

**Issue:** Several fire-and-forget `invoke` calls discard their Promise without a `.catch()`:

- `pty-bridge.ts:26`: `invoke('ack_bytes', { count: bytes.length, sessionName })` — no catch. If the Rust backend is busy or the PTY session dies, every incoming chunk will log an unhandled rejection.
- `state-manager.ts:147`: `invoke('save_state', { stateJson: ... }).catch(() => {})` — swallows all errors silently. This is acceptable for `beforeunload`, but the empty catch makes it impossible to diagnose state persistence failures.

**Fix:**
```ts
// pty-bridge.ts -- at minimum log ACK errors to avoid silent stream corruption
invoke('ack_bytes', { count: bytes.length, sessionName }).catch((e) => {
  console.warn('[efxmux] ack_bytes failed:', e);
});
```

### WR-02: Race Condition in PTY Session Fallback — `sessionName` Not Updated Before `attachResizeHandler`

**File:** `src/main.tsx:134-145`

**Issue:** When the primary PTY session fails, the code creates a `freshSession` name (`sessionName + '-new'`), calls `connectPty(terminal, freshSession)`, and calls `updateSession(...)`. However, `attachResizeHandler` on line 145 is always called with the original `sessionName` variable, not `freshSession`. If the fallback path is taken, all subsequent PTY resize calls go to the old (dead) session name:

```ts
// line 145 always uses the outer `sessionName`:
attachResizeHandler(container, terminal, fitAddon, sessionName);
```

**Fix:**
```ts
let activeSession = sessionName;
try {
  await connectPty(terminal, sessionName);
} catch (err) {
  // ... fallback ...
  activeSession = freshSession;
  await connectPty(terminal, freshSession);
  updateSession({ 'main-tmux-session': freshSession });
}
attachResizeHandler(container, terminal, fitAddon, activeSession);
```

### WR-03: `updateLayout` Side Effect Fires on Every Sidebar Expand/Collapse, Overwriting Drag-Set Width

**File:** `src/main.tsx:64-69`

**Issue:** The `effect()` at lines 64-69 runs whenever `sidebarCollapsed` changes. It unconditionally writes `sidebar-w` to either `'40px'` or `'200px'` and persists both values:

```ts
effect(() => {
  const collapsed = sidebarCollapsed.value;
  const w = collapsed ? '40px' : '200px';
  document.documentElement.style.setProperty('--sidebar-w', w);
  updateLayout({ 'sidebar-w': w, 'sidebar-collapsed': collapsed });
});
```

This overwrites any user-dragged sidebar width with `200px` on every expand. A user who resized the sidebar to 280px, collapses it, then expands it, will find it back at 200px. The dragged width in `state.json` is also overwritten.

**Fix:** Store the last non-collapsed width separately and restore it:
```ts
effect(() => {
  const collapsed = sidebarCollapsed.value;
  if (collapsed) {
    document.documentElement.style.setProperty('--sidebar-w', '40px');
    updateLayout({ 'sidebar-collapsed': true });
  } else {
    const savedW = getCurrentState()?.layout?.['sidebar-w'];
    const w = (savedW && savedW !== '40px') ? String(savedW) : '200px';
    document.documentElement.style.setProperty('--sidebar-w', w);
    updateLayout({ 'sidebar-w': w, 'sidebar-collapsed': false });
  }
});
```

### WR-04: `loadAppState` Returns a Non-Null Assertion on a Possibly-Null Value

**File:** `src/state-manager.ts:82`

**Issue:** `loadAppState()` ends with `return currentState!`. The non-null assertion suppresses the TypeScript error, but `currentState` is set to a hardcoded default in the catch block on line 67, so it is always non-null by the time of the return. However, if a future change removes the catch-branch default assignment, the `!` assertion will silently produce `undefined` at runtime. The safe pattern is to return the local variable from the catch block explicitly:

**Fix:**
```ts
export async function loadAppState(): Promise<AppState> {
  const defaults: AppState = {
    version: 1,
    layout: { 'sidebar-w': '200px', 'right-w': '25%', 'right-h-pct': '50', 'sidebar-collapsed': false },
    theme: { mode: 'dark' },
    session: { 'main-tmux-session': 'efx-mux', 'right-tmux-session': 'efx-mux-right' },
    project: { active: null },
    panels: { 'right-top-tab': 'gsd', 'right-bottom-tab': 'git' },
  };
  try {
    currentState = await invoke<AppState>('load_state');
  } catch (err) {
    console.warn('[efxmux] Failed to load state, using defaults:', err);
    currentState = defaults;
  }
  // ... signal setup ...
  return currentState; // no !
}
```

### WR-05: `handleProjectChanged` in `gsd-viewer.tsx` and `file-tree.tsx` Uses `setTimeout(fn, 50)` — Fragile Timing

**File:** `src/components/gsd-viewer.tsx:106-110`, `src/components/file-tree.tsx:70-75`

**Issue:** Both components wait 50ms after `project-changed` fires before reading the updated signal. This is necessary because `switchProject()` dispatches `project-changed` and then continues to set `activeProjectName.value` synchronously (they happen in order in `state-manager.ts:200-201`). The event fires before the signal is set. The 50ms delay masks this ordering issue. On a slow machine or under load, the signal may still reflect the old value when the timeout fires.

**Fix:** Dispatch the custom event **after** setting the signal, or — better — read the active project name from the event `detail` passed in the CustomEvent (which `switchProject` already provides):

```ts
// gsd-viewer.tsx
function handleProjectChanged(e: Event) {
  const name = (e as CustomEvent).detail?.name;
  const project = projects.value.find(p => p.name === name);
  if (project) loadGSD(project);
}
```

This removes the timeout and the ordering race.

### WR-06: `removeProject` in `sidebar.tsx` Does Not Guard Against Remove Failure

**File:** `src/components/sidebar.tsx:196-204`

**Issue:** The "Remove Project" button's `onClick` calls `removeProject(name)` without a try/catch:

```ts
onClick={async () => {
  await removeProject(name);
  removeTarget.value = null;
  const updatedProjects = await getProjects();
  projects.value = updatedProjects;
  const active = await getActiveProject();
  activeProjectName.value = active;
}}
```

If `removeProject` throws (e.g., Rust returns an error), `removeTarget.value` is never cleared, the dialog stays open, and the UI is stuck. Subsequent `getProjects()` and `getActiveProject()` calls also throw, leaving `projects.value` stale.

**Fix:**
```ts
onClick={async () => {
  try {
    await removeProject(name);
    removeTarget.value = null;
    const updatedProjects = await getProjects();
    projects.value = updatedProjects;
    const active = await getActiveProject();
    activeProjectName.value = active;
  } catch (err) {
    console.error('[efxmux] Failed to remove project:', err);
    removeTarget.value = null; // close dialog even on error
  }
}}
```

### WR-07: `loadDir` in `file-tree.tsx` Silently Ignores Parent Path Traversal Edge Case

**File:** `src/components/file-tree.tsx:112-119`

**Issue:** The Backspace handler computes the parent path by splitting on `/` and rejoining. On macOS, paths always start with `/`, so splitting `/foo/bar` gives `['', 'foo', 'bar']` and `slice(0, -1).join('/')` gives `/foo` — correct. However, `currentPath.value` at the project root would split `/Users/x/project` and produce `/Users/x`. The check `parent.startsWith(rootPath)` would then be false (since `rootPath` is `/Users/x/project`), correctly blocking navigation up. But if `rootPath` has a trailing slash (`/Users/x/project/`), `startsWith` would fail for the correct parent path `/Users/x`. The CLAUDE.md notes that `path` comes from the OS dialog which on macOS does not append trailing slashes, but this assumption is not validated.

**Fix:** Normalize both paths before comparing:
```ts
const normalize = (p: string) => p.replace(/\/+$/, '');
if (parent && rootPath && normalize(parent).startsWith(normalize(rootPath))) {
  loadDir(parent);
}
```

### WR-08: `injectLineNumbers` in `gsd-viewer.tsx` Mutates Rendered HTML via Fragile Regex

**File:** `src/components/gsd-viewer.tsx:33-43`

**Issue:** `injectLineNumbers` uses a regex to find `<input type="checkbox" class="task-checkbox"...>` and inject `data-line` attributes. The regex relies on exact attribute order from `marked`'s custom renderer. If `marked` ever emits attributes in a different order (e.g., `class` before `type`), or adds whitespace, the regex will fail silently — checkboxes will render without `data-line`, and click-to-toggle will be a no-op (line 133: `if (isNaN(line) || line < 0) return`).

**Fix:** Use a more permissive regex or a DOM parser, or use a `marked` extension that can inject the attribute at render time (avoiding post-processing entirely):

```ts
// In the custom renderer, pass line number via a closure that increments an index
// tracked during a two-pass parse, then use data-line directly in the renderer.
// Alternatively, use a permissive regex:
return renderedHtml.replace(
  /<input\s+[^>]*class="task-checkbox"[^>]*>/g,
  (match) => {
    const line = lineMap[checkboxIndex] !== undefined ? lineMap[checkboxIndex] : -1;
    checkboxIndex++;
    return match.replace('class="task-checkbox"', `class="task-checkbox" data-line="${line}"`);
  }
);
```

### IN-01: Missing `key` Props on Mapped Lists (Preact/React Convention)

**File:** `src/components/sidebar.tsx:289,323`, `src/components/file-tree.tsx:136`, `src/components/fuzzy-search.tsx:205`

**Issue:** Several `.map()` calls render lists without `key` props. Preact uses keys to reconcile list updates efficiently. Without them, all list items re-render on any state change, and if items are reordered, DOM state (focus, scroll position) will be misassigned.

```tsx
// sidebar.tsx -- missing keys
projects.value.map((p, i) => <CollapsedIcon project={p} index={i} />)
projects.value.map((p, i) => <ProjectRow project={p} index={i} />)
gitFiles.value.map(f => <GitFileRow file={f} />)
```

**Fix:** Add stable keys — `project.name` for projects (unique by validation), `file.path` for git files:
```tsx
projects.value.map((p, i) => <ProjectRow key={p.name} project={p} index={i} />)
gitFiles.value.map(f => <GitFileRow key={f.path} file={f} />)
results.map((p, i) => <SearchResult key={p.name} project={p} index={i} />)
```

### IN-05: `tauri.conf.json` CSP Uses `unsafe-inline` for Both Scripts and Styles

**File:** `src-tauri/tauri.conf.json:24`

**Issue:** The CSP is `"default-src 'self'; script-src 'self' 'unsafe-inline'; style-src 'self' 'unsafe-inline'"`. The `unsafe-inline` for `script-src` is present. While Tauri's WKWebView applies additional sandboxing beyond the CSP, `unsafe-inline` scripts would execute if XSS were achieved via `innerHTML` injection (as in CR-01). This is a defense-in-depth concern rather than a standalone vulnerability.

**Fix:** Consider using a nonce-based CSP or removing `unsafe-inline` from `script-src` if Vite's production build can be configured to avoid inline scripts. Tauri 2 supports nonce-based CSP via the `nonce` configuration option.

**In src/main.tsx -- keyboard handler (line 106):**

The guard clause `if (!e.ctrlKey || e.metaKey) return` blocks ALL Cmd-based shortcuts. Cmd+W never reaches closeActiveTab.

Replace the guard clause and the `case key === 'w'` block:

1. Change the guard at line 106 from:
   ```typescript
   if (!e.ctrlKey || e.metaKey) return;
   ```
   to:
   ```typescript
   // Allow both Ctrl+key and Cmd+key for 'w' (close tab)
   if (!e.ctrlKey && !e.metaKey) return;
   ```

2. Update the `case key === 'w'` block to handle both Ctrl+W and Cmd+W:
   ```typescript
   case key === 'w' && !e.shiftKey && !e.altKey && (e.ctrlKey || e.metaKey):
     e.preventDefault(); e.stopPropagation();
     closeActiveTab();
     break;
   ```

3. All OTHER shortcut cases must still require `e.ctrlKey` explicitly (not metaKey), so add `&& e.ctrlKey` to each non-W case. The existing cases already match on ctrlKey implicitly via the old guard, but now that metaKey passes the guard too, we must be explicit. Update each case:
   - `case key === 'b' && e.ctrlKey && !e.shiftKey && !e.altKey:`
   - `case key === 's' && e.ctrlKey && !e.shiftKey && !e.altKey:`
   - `case key === 't' && e.ctrlKey && !e.shiftKey && !e.altKey:`
   - `case e.key === 'Tab' && e.ctrlKey && !e.shiftKey && !e.altKey:` (already has e.ctrlKey)
   - `case key === 'p' && e.ctrlKey && !e.shiftKey && !e.altKey:`
   - And all other existing cases -- add `&& e.ctrlKey` guard.

4. Also keep the terminal passthrough check but scope it to ctrlKey only:
   ```typescript
   if (e.ctrlKey && TERMINAL_PASSTHROUGH.has(key) && !e.shiftKey && !e.altKey) return;
   ```

**In src-tauri/src/lib.rs -- Window menu (lines 46-49):**

Remove `PredefinedMenuItem::close_window` which binds Cmd+W at the native level (fires before JS can intercept it).

Change the window_menu from:
```rust
let window_menu = SubmenuBuilder::new(app, "Window")
    .item(&PredefinedMenuItem::minimize(app, None)?)
    .item(&PredefinedMenuItem::close_window(app, None)?)
    .build()?;
```
to:
```rust
let window_menu = SubmenuBuilder::new(app, "Window")
    .item(&PredefinedMenuItem::minimize(app, None)?)
    .build()?;
```

This removes the native Cmd+W accelerator so the JS handler can intercept it.

**Bug 1 - Wizard/sidebar race (sidebar.tsx line 235-236):**

Both `initProjects()` in main.tsx and the sidebar's `useEffect` independently detect 0 projects. The sidebar calls `openProjectModal()` which shows Add Project modal, competing with the wizard.

Fix: In sidebar.tsx, REMOVE the `openProjectModal()` call from the useEffect that loads projects. Around line 235-236, remove:
```typescript
if (loadedProjects.length === 0) {
  openProjectModal();
}
```

The wizard in main.tsx's `initProjects()` already handles zero-project detection correctly. The sidebar should only display the project list, not trigger modals.

**Bug 2 - AppState missing projects field (state-manager.ts lines 29-36, 68-76):**

The `AppState` interface in state-manager.ts does not include a `projects` field. When `save_state` is called, it serializes the JS AppState which lacks project data. This overwrites the Rust AppState (which DOES have projects) with a version missing them.

Fix in state-manager.ts:
1. Add `projects` to the AppState interface:
   ```typescript
   export interface AppState {
     version: number;
     layout: Record<string, string | boolean>;
     theme: { mode: string };
     session: Record<string, string>;
     project: { active: string | null };
     panels: Record<string, string>;
     projects: ProjectEntry[];  // ADD THIS
   }
   ```

2. In the catch-block default state (around line 68-76), add `projects: []`:
   ```typescript
   currentState = {
     version: 1,
     layout: { ... },
     theme: { mode: 'dark' },
     session: { ... },
     project: { active: null },
     panels: { ... },
     projects: [],  // ADD THIS
   };
   ```

3. In the `saveState` / `initBeforeUnload` logic, ensure `currentState.projects` is populated from the `projects` signal before serializing:
   - Find where `save_state` is called (likely in `initBeforeUnload` or a periodic save).
   - Before the `invoke('save_state', { stateJson: JSON.stringify(currentState) })` call, sync the projects signal:
     ```typescript
     if (currentState) {
       currentState.projects = projects.value;
       currentState.project = { active: activeProjectName.value };
     }
     ```

4. In `loadAppState()`, after loading state, populate the projects signal from loaded state:
   ```typescript
   if (currentState?.projects?.length) {
     projects.value = currentState.projects;
   }
   ```

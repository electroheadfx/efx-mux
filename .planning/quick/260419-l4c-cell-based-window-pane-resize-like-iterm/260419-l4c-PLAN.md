<objective>
Eliminate the xterm row/column remainder band by making the macOS window AND all internal split handles snap to character-cell boundaries — iTerm2 / Ghostty parity.

**Architecture C from RESEARCH.md Section 2**: enable NSWindow contentResizeIncrements ONLY when the main column's active tab is a terminal. For non-terminal tabs (editor, file-tree, gsd, git-changes), clear increments to (1,1) so free pixel resize is restored. Internal handles use JS-side live onDrag + onEnd quantization via a shared `snapToCell` helper — drag-level only, NEVER render-level.

**CRITICAL anti-pattern the prior k1n attempt hit**: DO NOT apply `inline height: Npx + flex: none` to `.sub-scope-pane` at render-time. Window-level NSWindow snap + drag-level `snapToCell` replaces pane-pinning entirely. The existing intra-zone drag already mutates pane.style.height on drag (22-11 gap-closure); we layer `snapToCell` on top of that — we do NOT change its lifecycle.

Purpose: matches the terminal-first aesthetic of Efxmux (iTerm2/Ghostty parity), kills the partial-cell remainder band the user has been chasing for 7+ commits on Phase 22.

Output:
- Two new Rust commands (+ objc2 crates) gated to macOS, no-op stubs elsewhere.
- One new TS module (resize-increments.ts) with unit tests.
- Six wiring sites in drag-manager, terminal-manager, theme-manager, main.tsx.
</objective>

**Implementation notes**:

1. `getActiveTerminalCellGeom()`:
   - Query `document.querySelector<HTMLElement>('.main-panel .xterm:not([style*="display: none"])')` (per RESEARCH §3). The `:not([style*="display: none"])` filter is essential because SubScopePane always-mounts bodies with `display: none` toggled — we want the visible terminal.
   - If no element found → return null (startup, or non-terminal active tab).
   - Access `_core` via `(el as any)._xterm?._core ?? (el as any).terminal?._core` (xterm exposes the instance on the container element after `.open()`). Prefer the first form; the fallback exists because Preact-wrapped containers have sometimes differed. Use `// eslint-disable-next-line @typescript-eslint/no-explicit-any`.
   - Read `core?._renderService?.dimensions?.css?.cell` → if missing or `width &lt; 1` or `height &lt; 1` return null.
   - `rect = el.getBoundingClientRect()` → return `{ originX: rect.left, originY: rect.top, cellW: d.width, cellH: d.height }`.

2. `snapToCell(px, axis)` — pure function (unit-testable):
   - Call `getActiveTerminalCellGeom()`; if null → return `px` (pass-through).
   - `origin = axis === 'x' ? g.originX : g.originY`
   - `step = axis === 'x' ? g.cellW : g.cellH`
   - Guard: `if (step < 1) return px;`
   - `offset = px - origin; snappedOffset = Math.round(offset / step) * step; return origin + snappedOffset;`
   - CRITICAL: DO NOT return `origin` as a fallback — that collapses the handle to the pane edge (pitfall from RESEARCH §5).

3. `syncWindowIncrements(cellW, cellH)`:
   - `await invoke('set_content_resize_increments', { cellW, cellH }).catch((e: unknown) => console.warn('[efxmux] set_content_resize_increments failed:', e))`
   - Do NOT throw on IPC failure — increments are a nice-to-have, not a correctness requirement.

4. `syncIncrementsDebounced()`:
   - Module-scoped `setTimeout` handle, 100ms trailing edge.
   - On fire: call `getActiveTerminalCellGeom()`; if null return without IPC (no terminal → nothing to sync).
   - Round cellW/cellH to nearest 0.5 before passing (per RESEARCH §5 sub-pixel pitfall): `Math.round(cellW * 2) / 2`.

5. `clearWindowIncrements()`:
   - `await invoke('clear_content_resize_increments').catch(() => {})` — silent fail (non-macOS stub always Ok).

<task type="auto">
  <name>Task 1: Replace remain-on-exit EOF strategy with pane-death monitoring thread</name>
  <files>src-tauri/src/terminal/pty.rs, src-tauri/src/lib.rs</files>
  <action>
The current approach (lines 106-109) sets remain-on-exit=on, expecting the read loop to eventually EOF and then query pane_dead_status. But remain-on-exit keeps the PTY master alive, so EOF never happens and lines 183-229 are unreachable.

Fix strategy:
1. KEEP remain-on-exit=on (needed to query exit code after death).
2. REMOVE the post-EOF exit detection block (lines 183-229 inside the read loop thread).
3. After spawning the read loop thread, spawn a SECOND monitoring thread that:
   a. Sleeps 1 second initially (let tmux stabilize).
   b. Polls `tmux display-message -t {session} -p "#{pane_dead}"` every 500ms.
   c. When `#{pane_dead}` returns `1`:
      - Query `#{pane_dead_status}` for the real exit code.
      - Kill the tmux session (`tmux kill-session -t {session}`).
      - Emit `pty-exited` event with `{ session, code }`.
      - Break the monitoring loop.
   d. Also break if `tmux has-session -t {session}` returns failure (session already gone) -- emit exit code 0.
   e. Store a `Arc<AtomicBool>` "stopped" flag shared with the read loop. When monitoring detects death, set stopped=true. The read loop should also check this flag periodically (e.g., after each read timeout) to break out.

4. Add a `cleanup_dead_sessions` Tauri command that:
   - Runs `tmux list-sessions -F "#{session_name}:#{pane_dead}"`.
   - For any session with pane_dead=1, kills it via `tmux kill-session`.
   - Called from JS on app startup (will be wired in Plan 06 task).
   - Export via `pub fn cleanup_dead_sessions()`.

5. Register `cleanup_dead_sessions` in lib.rs invoke_handler list.

The read loop thread (lines 146-181) stays mostly as-is but:
- Remove lines 183-229 (unreachable exit detection).
- The loop already breaks on EOF or error, which is fine -- the monitoring thread handles exit detection independently.

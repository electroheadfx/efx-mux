<task type="auto">
  <name>Task 1: Move kill_all_servers to RunEvent::ExitRequested handler</name>
  <files>src-tauri/src/lib.rs</files>
  <action>
    **Root cause:** `on_window_event(CloseRequested)` only fires when the user clicks the window close button (red dot). Cmd+Q dispatches through the macOS application menu "Quit" item, which triggers `RunEvent::ExitRequested` at the App level, NOT a window event. The current kill_all_servers call in on_window_event never executes on Cmd+Q.

    **Fix:** Replace the `.run(tauri::generate_context!()).expect(...)` pattern with `.build(tauri::generate_context!()).expect(...).run(callback)` to intercept `RunEvent::ExitRequested`.

    In `src-tauri/src/lib.rs`:

    1. Change the end of the builder chain from:
       ```rust
       .run(tauri::generate_context!())
       .expect("error while running tauri application")
       ```
       to:
       ```rust
       .build(tauri::generate_context!())
       .expect("error while building tauri application")
       .run(|app_handle, event| {
           if let tauri::RunEvent::ExitRequested { .. } = &event {
               kill_all_servers(app_handle);
               // Save state on quit (same as close handler)
               let managed = app_handle.state::<ManagedAppState>();
               let snapshot = managed.0.lock().ok().map(|g| g.clone());
               if let Some(ref s) = snapshot {
                   if let Err(e) = state::save_state_sync(s) {
                       eprintln!("[efxmux] WARNING: Failed to save state on quit: {}", e);
                   }
               }
           }
       });
       ```

    2. Keep the existing `on_window_event(CloseRequested)` handler as-is. It handles the window close button case AND saves state. The RunEvent handler handles Cmd+Q. Both paths call kill_all_servers and save state -- this is intentional redundancy since only one path fires per exit method.

    3. Add `use tauri::Manager;` if not already present (it is -- line 7). Verify `state::save_state_sync` and `ManagedAppState` are accessible (they are -- imported at line 18).

    **Note on log persistence (UAT gap 4 secondary concern):** The user reported "Logs of each project are cleared, not persistent." Server logs are in-memory Preact signals and are intentionally ephemeral -- they come from live process stdout. Persisting them to disk is a feature enhancement, not a bug fix. The critical issue (blocker) is that processes survive app exit. Log persistence is deferred as it would require a new file I/O subsystem (log rotation, disk cleanup, etc.) that is out of scope for gap closure.

<task type="auto">
  <name>Task 2: Fix ANSI color rendering with line-buffered emission</name>
  <files>src-tauri/src/server.rs</files>
  <action>
    **Root cause:** The Rust stdout/stderr reader threads read in 4096-byte raw buffers and emit immediately. ANSI escape sequences (e.g., `\x1b[32m`) can be split across read() boundaries -- the first read sends `\x1b[3` and the next sends `2mHello`. The JS regex `\x1b\[(\d+(?:;\d+)*)m` never matches partial sequences. Three prior fix attempts addressed CSS specificity and regex patterns but never addressed the split-sequence root cause.

    **Fix:** Add line buffering to the stdout and stderr reader threads in `src-tauri/src/server.rs` so that each emitted event contains complete lines (which contain complete ANSI sequences).

    In the stdout reader thread (lines 77-111):
    1. Replace the raw `[0u8; 4096]` buffer + `read()` loop with a `BufReader` + line-based reading.
    2. Add `use std::io::BufRead;` at the top of server.rs (alongside existing `use std::io::Read;`).
    3. Replace the reader loop with:
       ```rust
       let reader = std::io::BufReader::new(stdout);
       for line in reader.lines() {
           match line {
               Ok(text) => {
                   let text = text + "\n"; // Restore newline stripped by lines()
                   let payload = serde_json::json!({ "project": project_id_clone, "text": text });
                   let _ = app_clone.emit("server-output", payload);
               }
               Err(_) => break,
           }
       }
       ```
       The `BufReader::lines()` iterator reads until `\n`, ensuring ANSI sequences within a line are never split. The `+ "\n"` restores the newline that `lines()` strips, which the frontend `ansiToHtml` converts to `<br>`.

    4. Apply the exact same change to the stderr reader thread (lines 115-154).

    5. Keep the EOF-based exit detection logic after both loops (the `reader_count.fetch_sub` + `waitpid` pattern) -- it works correctly and just needs to stay after the loop.

    **Important:** Do NOT change ansi-html.ts. The JS converter works correctly when it receives complete ANSI sequences. The bug is in the emission granularity, not the parsing.
  </action>

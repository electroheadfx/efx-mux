The current code at lines 284-289 is:
```typescript
const newMainSession = projectSessionName(newProjectName);
if (newMainSession !== mainCurrentSession) {
  await switchTmuxSession(mainCurrentSession, newMainSession, project.path);
  mainCurrentSession = newMainSession;

```

Replace that block with agent-aware session switching. The `switchTmuxSession` function switches the underlying tmux session silently (no PTY reconnect). But to launch an agent in the new session, we need to detect the agent binary and pass it when connecting. The key insight: `switchTmuxSession` handles existing sessions. For a NEW session that needs an agent, the agent binary must be passed.

Updated logic for the main terminal section inside the `project-changed` listener:

```typescript
const newMainSession = projectSessionName(newProjectName);
if (newMainSession !== mainCurrentSession) {
  // Detect agent for the new project (AGENT-03, AGENT-04)
  let agentBinary: string | null = null;
  if (project.agent && project.agent !== 'bash') {
    try {
      agentBinary = await detectAgent(project.agent);
    } catch {
      agentBinary = null;
    }
  }

  // Switch tmux session (creates if needed, agent passed for new sessions)
  await switchTmuxSession(mainCurrentSession, newMainSession, project.path);
  mainCurrentSession = newMainSession;
}
```

IMPORTANT: The `switchTmuxSession` function calls `invoke('switch_tmux_session', ...)` which in `pty.rs` runs `tmux switch-client`. This switches an already-attached client to a different session. If the target session doesn't exist yet, tmux creates it with `new-session -d -s {target} -c {startDir}` first.

The issue is that when a NEW session is created by `switch_tmux_session` in Rust, it uses the default shell (bash) -- it does NOT know about the agent binary. The agent binary is only passed during `spawn_terminal` (initial connect).

To fix this properly, we need to also pass the agent's shell command when switching. Check if `switch_tmux_session` in `pty.rs` accepts a `shell_command` parameter. If not, we need to update the Rust side too.

**Check `src-tauri/src/terminal/pty.rs` for the `switch_tmux_session` function signature.** If it does NOT accept `shell_command`:

1. Add `shell_command: Option<String>` parameter to the `switch_tmux_session` Rust command
2. When creating a new session (the `tmux new-session -d -s {target}` path), append the shell_command to the args if Some
3. Update the `switchTmuxSession` function in `src/main.tsx` (lines 53-58) to accept and pass `shellCommand?: string`

Then update the project-changed listener call:
```typescript
await switchTmuxSession(mainCurrentSession, newMainSession, project.path, agentBinary ?? undefined);
```

If the Rust `switch_tmux_session` command ALREADY accepts `shell_command`, just pass it through from the JS side.

**Fallback: If modifying the Rust side is complex**, an alternative approach that works without Rust changes:
- After `switchTmuxSession`, if an agent binary was detected and the session is new, send the agent command to the tmux session via `invoke('write_pty', { data: agentBinary + '\n', sessionName: newMainSession })` or equivalent PTY write command. This types the agent command into the bash session. Less clean but functional.

Choose the approach that compiles cleanly. The goal is: when switching to a project with agent="claude", if the claude binary exists, the new tmux session runs claude (not just bash).
